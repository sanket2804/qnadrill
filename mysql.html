🟢 Basics — Q1 to Q12

Core MySQL fundamentals every developer must know

Q1 What is MySQL? What are its key features? ▼

MySQL is an open-source relational database management system (RDBMS) that uses Structured Query Language (SQL). It is one of the world's most popular databases, used by companies like Facebook, Twitter, YouTube, and Airbnb.

  • Relational model — data stored in structured tables with rows and columns
  • ACID compliance — Atomicity, Consistency, Isolation, Durability (with InnoDB engine)
  • Multiple storage engines — InnoDB, MyISAM, MEMORY, CSV, etc.
  • Cross-platform — runs on Windows, Linux, macOS
  • Replication support — master-slave and master-master replication
  • Full-text search — FULLTEXT indexes for text search
  • Stored procedures, triggers, views — server-side programming
  • Partitioning — split large tables across storage for performance
Interview tip: MySQL 8.0 (2018) added window functions, CTEs, roles, and many PostgreSQL-like features. Always mention the version when discussing features.
Q2 What is the difference between DDL, DML, DCL, and TCL? ▼
  • DDL (Data Definition Language): Defines/modifies database structure. Auto-commits. Commands: CREATE, ALTER, DROP, TRUNCATE, RENAME
  • DML (Data Manipulation Language): Manipulates data in tables. Can be rolled back. Commands: SELECT, INSERT, UPDATE, DELETE
  • DCL (Data Control Language): Controls access and permissions. Commands: GRANT, REVOKE
  • TCL (Transaction Control Language): Manages transactions. Commands: COMMIT, ROLLBACK, SAVEPOINT, SET TRANSACTION
-- DDL
CREATE TABLE users (id INT PRIMARY KEY, name VARCHAR(100));
ALTER TABLE users ADD COLUMN email VARCHAR(255);

-- DML
INSERT INTO users (id, name) VALUES (1, 'Alice');
UPDATE users SET name = 'Bob' WHERE id = 1;

-- DCL
GRANT SELECT, INSERT ON mydb.* TO 'dev'@'localhost';
Q3 What are MySQL data types? Explain the most important ones. ▼
  • Numeric: INT, TINYINT, BIGINT, FLOAT, DOUBLE, DECIMAL(p,s) — use DECIMAL for money!
  • String: CHAR(n) — fixed length, VARCHAR(n) — variable length, TEXT — large text, ENUM, SET
  • Date/Time: DATE (YYYY-MM-DD), TIME, DATETIME, TIMESTAMP (auto timezone), YEAR
  • Binary: BLOB, BINARY, VARBINARY
  • JSON: Native JSON type (MySQL 5.7.8+)
CREATE TABLE products (
  id         INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  name       VARCHAR(200) NOT NULL,
  price      DECIMAL(10, 2) NOT NULL,   -- e.g. 99999999.99
  status     ENUM('active','inactive') DEFAULT 'active',
  meta       JSON,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
Never use FLOAT/DOUBLE for money — floating point imprecision causes errors. Always use DECIMAL.
Q4 What is the difference between CHAR and VARCHAR? ▼
  • CHAR(n): Fixed-length. Always uses n bytes (pads with spaces). Faster for fixed-length data. Max 255 chars.
  • VARCHAR(n): Variable-length. Uses actual length + 1-2 bytes overhead. More space-efficient. Max 65,535 bytes (row limit).
-- CHAR: always 10 bytes
country_code CHAR(2)    -- 'IN' always 2 bytes
phone_ext    CHAR(5)    -- '123' stored as '123  '

-- VARCHAR: variable
email        VARCHAR(255) -- 'a@b.com' = 7+1 bytes
description  VARCHAR(1000)
Use CHAR for fixed-length data (country codes, UUIDs, status flags). Use VARCHAR for variable-length data (names, emails, descriptions).
Q5 What is the difference between DELETE, TRUNCATE, and DROP? ▼
  • DELETE: DML — removes specific rows (with WHERE). Can be rolled back. Fires triggers. Slow for large datasets. AUTO_INCREMENT not reset.
  • TRUNCATE: DDL — removes ALL rows. Cannot be rolled back (auto-commits). Does NOT fire triggers. Fast (resets storage). AUTO_INCREMENT resets.
  • DROP: DDL — removes the entire table structure AND data. Cannot be rolled back. Table no longer exists.
DELETE FROM orders WHERE status = 'cancelled'; -- selective, rollback-able
TRUNCATE TABLE temp_logs;   -- wipe all, fast, auto-commits
DROP TABLE old_reports;     -- remove table entirely
Q6 What are constraints in MySQL? ▼

Constraints enforce rules on data in tables, ensuring data integrity.

  • PRIMARY KEY: Uniquely identifies each row. NOT NULL + UNIQUE. One per table.
  • FOREIGN KEY: Links to PRIMARY KEY in another table. Enforces referential integrity.
  • UNIQUE: Ensures all values in a column are distinct. Allows NULL (multiple NULLs allowed).
  • NOT NULL: Column cannot contain NULL values.
  • CHECK: Ensures values satisfy a condition (MySQL 8.0.16+).
  • DEFAULT: Sets a default value when none is provided.
CREATE TABLE orders (
  id         INT AUTO_INCREMENT PRIMARY KEY,
  user_id    INT NOT NULL,
  amount     DECIMAL(10,2) CHECK (amount > 0),
  email      VARCHAR(255) UNIQUE,
  status     VARCHAR(20) DEFAULT 'pending',
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);
Q7 What is the difference between InnoDB and MyISAM storage engines? ▼
  • InnoDB (default since MySQL 5.5): ACID compliant, supports transactions, foreign keys, row-level locking, crash recovery, MVCC. Best for OLTP (most apps).
  • MyISAM (legacy): No transactions, no foreign keys, table-level locking only. Supports FULLTEXT index. Fast for read-heavy, non-critical data. Not crash-safe.
CREATE TABLE logs (
  id  INT PRIMARY KEY,
  msg TEXT
) ENGINE=InnoDB;   -- transactions, FK support

SHOW TABLE STATUS LIKE 'users';  -- shows engine type
ALTER TABLE myisam_table ENGINE=InnoDB;  -- convert engine
Always use InnoDB unless you have a specific reason. MyISAM is largely obsolete for new development.
Q8 What are NULL values in MySQL? How do you handle them? ▼

NULL means "unknown" or "missing" — it is NOT zero, empty string, or false. Any comparison with NULL returns NULL (not true/false), which is why you must use IS NULL / IS NOT NULL.

-- Wrong: returns no rows even if name IS NULL
SELECT * FROM users WHERE name = NULL;

-- Correct
SELECT * FROM users WHERE name IS NULL;
SELECT * FROM users WHERE name IS NOT NULL;

-- COALESCE: return first non-NULL value
SELECT COALESCE(nickname, first_name, 'Anonymous') FROM users;

-- IFNULL: return alternative if NULL
SELECT IFNULL(phone, 'N/A') FROM users;

-- NULLIF: return NULL if two values are equal
SELECT NULLIF(score, 0) FROM results; -- returns NULL instead of 0
Q9 What is AUTO_INCREMENT in MySQL? ▼

AUTO_INCREMENT automatically generates a unique numeric value for a column (typically the primary key) each time a new row is inserted.

CREATE TABLE users (
  id   INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  name VARCHAR(100)
);

INSERT INTO users (name) VALUES ('Alice'), ('Bob');
-- Alice gets id=1, Bob gets id=2

-- Get last inserted ID
SELECT LAST_INSERT_ID();

-- Reset auto_increment
ALTER TABLE users AUTO_INCREMENT = 1000;

-- Check current value
SHOW TABLE STATUS LIKE 'users';
AUTO_INCREMENT values are never reused — deleting a row does not recycle its ID (except after TRUNCATE).
Q10 What is the difference between HAVING and WHERE? ▼
  • WHERE: Filters rows BEFORE grouping. Cannot use aggregate functions (SUM, COUNT, etc.).
  • HAVING: Filters groups AFTER GROUP BY. CAN use aggregate functions.
-- WHERE filters rows before grouping
SELECT department, COUNT(*) AS emp_count
FROM employees
WHERE status = 'active'        -- filters rows first
GROUP BY department
HAVING COUNT(*) > 5;           -- then filters groups

-- Find departments with total salary > 100000
SELECT department, SUM(salary) AS total
FROM employees
GROUP BY department
HAVING SUM(salary) > 100000;
Q11 What are aggregate functions in MySQL? ▼

Aggregate functions perform calculations on a set of rows and return a single value. They are used with GROUP BY.

SELECT
  COUNT(*)           AS total_rows,      -- all rows including NULLs
  COUNT(email)       AS with_email,      -- NULLs excluded
  COUNT(DISTINCT city) AS unique_cities,
  SUM(amount)        AS total_sales,
  AVG(salary)        AS avg_salary,
  MIN(created_at)    AS oldest,
  MAX(created_at)    AS newest,
  GROUP_CONCAT(name ORDER BY name SEPARATOR ', ') AS names
FROM orders;
Q12 What is the order of SQL query execution? ▼

SQL clauses are written in one order but executed in a different order. Understanding this prevents confusion about why aliases aren't available in WHERE.

-- Written order:
SELECT department, COUNT(*) AS cnt
FROM employees
WHERE status = 'active'
GROUP BY department
HAVING cnt > 5
ORDER BY cnt DESC
LIMIT 10;

Execution order:

  • 1. FROM / JOIN — identify tables
  • 2. WHERE — filter rows
  • 3. GROUP BY — group rows
  • 4. HAVING — filter groups
  • 5. SELECT — select columns, compute expressions
  • 6. DISTINCT — remove duplicates
  • 7. ORDER BY — sort results
  • 8. LIMIT / OFFSET — paginate

🔵 Queries & Joins — Q13 to Q26

SELECT queries, all types of JOINs, subqueries, and set operations

Q13 What are the different types of JOINs in MySQL? ▼
  • INNER JOIN: Returns rows where there is a match in BOTH tables.
  • LEFT JOIN (LEFT OUTER): Returns ALL rows from left table + matched rows from right. NULLs for no match.
  • RIGHT JOIN (RIGHT OUTER): Returns ALL rows from right table + matched rows from left. NULLs for no match.
  • CROSS JOIN: Cartesian product — every row from left paired with every row from right.
  • SELF JOIN: A table joined with itself (using aliases).
-- INNER JOIN: only matching rows
SELECT u.name, o.amount
FROM users u
INNER JOIN orders o ON u.id = o.user_id;

-- LEFT JOIN: all users, even without orders
SELECT u.name, o.amount
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;

-- Find users with NO orders
SELECT u.name
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.id IS NULL;
Q14 What is a SELF JOIN? Give a practical example. ▼

A self join joins a table to itself using two different aliases. Useful for hierarchical data like organizational structures or finding related rows in the same table.

-- employees table has: id, name, manager_id (references id)
SELECT
  e.name        AS employee,
  m.name        AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;
-- Alice | Bob (Alice reports to Bob)
-- Bob   | NULL (Bob has no manager = CEO)

-- Find employees in same department
SELECT a.name, b.name, a.department
FROM employees a
JOIN employees b ON a.department = b.department
WHERE a.id < b.id; -- avoid duplicates and self-pairing
Q15 What is a subquery? What are its types? ▼

A subquery is a query nested inside another query. Types:

  • Scalar subquery: Returns a single value — used in SELECT or WHERE
  • Row subquery: Returns a single row with multiple columns
  • Column subquery: Returns a single column — used with IN, ANY, ALL
  • Table subquery: Returns a virtual table — used in FROM clause
  • Correlated subquery: References the outer query — executed once per outer row
-- Scalar: single value in WHERE
SELECT * FROM orders
WHERE amount > (SELECT AVG(amount) FROM orders);

-- Column: IN with subquery
SELECT name FROM users
WHERE id IN (SELECT user_id FROM orders WHERE status='paid');

-- Correlated: references outer query
SELECT name,
  (SELECT COUNT(*) FROM orders o WHERE o.user_id = u.id) AS order_count
FROM users u;
Q16 What is the difference between IN, EXISTS, and ANY/ALL? ▼
  • IN: Checks if value matches any item in a list or subquery result. The subquery runs once.
  • EXISTS: Returns true if the subquery returns at least one row. Stops scanning as soon as a match is found — often faster for large datasets.
  • ANY / ALL: Compares a value with a set. ANY = at least one, ALL = all values must satisfy.
-- IN (subquery runs once)
SELECT * FROM users WHERE id IN (SELECT user_id FROM orders);

-- EXISTS (faster for large sets — short circuits)
SELECT * FROM users u
WHERE EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id);

-- ANY/ALL
SELECT * FROM products
WHERE price > ANY (SELECT price FROM products WHERE category='A');
-- greater than at least one price in category A
Q17 What is a Common Table Expression (CTE)? ▼

A CTE (introduced in MySQL 8.0) defines a temporary named result set that you can reference multiple times in the same query. Makes complex queries more readable than nested subqueries. Use WITH keyword.

-- Simple CTE
WITH high_value_orders AS (
  SELECT user_id, SUM(amount) AS total
  FROM orders
  GROUP BY user_id
  HAVING total > 1000
)
SELECT u.name, hvo.total
FROM users u
JOIN high_value_orders hvo ON u.id = hvo.user_id;

-- Multiple CTEs
WITH
  active_users AS (SELECT * FROM users WHERE status='active'),
  recent_orders AS (SELECT * FROM orders WHERE created_at > '2024-01-01')
SELECT u.name, o.amount
FROM active_users u JOIN recent_orders o ON u.id = o.user_id;
Q18 What are Recursive CTEs? ▼

Recursive CTEs reference themselves to traverse hierarchical data (org charts, category trees, bills of materials) without knowing the depth in advance.

-- employees: id, name, manager_id
WITH RECURSIVE org_chart AS (
  -- Base case: top-level (CEO)
  SELECT id, name, manager_id, 0 AS level
  FROM employees WHERE manager_id IS NULL

  UNION ALL

  -- Recursive case: employees reporting to previous level
  SELECT e.id, e.name, e.manager_id, oc.level + 1
  FROM employees e
  JOIN org_chart oc ON e.manager_id = oc.id
)
SELECT CONCAT(REPEAT('  ', level), name) AS hierarchy
FROM org_chart;
Q19 What are Window Functions in MySQL? ▼

Window functions (MySQL 8.0+) perform calculations across a set of rows related to the current row without collapsing them into groups. They use the OVER() clause.

-- ROW_NUMBER: unique rank per partition
SELECT name, department, salary,
  ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) AS dept_rank,
  RANK()       OVER (ORDER BY salary DESC) AS overall_rank,
  DENSE_RANK() OVER (ORDER BY salary DESC) AS dense_rank,
  LAG(salary)  OVER (ORDER BY salary)        AS prev_salary,
  SUM(salary)  OVER (PARTITION BY department) AS dept_total
FROM employees;

-- Top 1 salary per department
SELECT * FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC) rn
  FROM employees
) ranked WHERE rn = 1;
Q20 What is the difference between UNION and UNION ALL? ▼
  • UNION: Combines results of two SELECT statements, removing duplicates. Slower (requires sorting/hashing to find duplicates).
  • UNION ALL: Combines results including duplicates. Faster — no deduplication.
-- UNION: unique emails from both tables
SELECT email FROM customers
UNION
SELECT email FROM suppliers;

-- UNION ALL: all emails including duplicates (faster)
SELECT email, 'customer' AS type FROM customers
UNION ALL
SELECT email, 'supplier' AS type FROM suppliers;
Use UNION ALL by default. Only switch to UNION when you specifically need deduplication.
Q21 What is the difference between RANK(), DENSE_RANK(), and ROW_NUMBER()? ▼
  • ROW_NUMBER(): Sequential numbers — no ties. Every row gets a unique number.
  • RANK(): Same rank for ties, but skips numbers after ties (1,1,3 — skips 2).
  • DENSE_RANK(): Same rank for ties, no gaps (1,1,2 — no skip).
-- Given salaries: 100, 90, 90, 80
SELECT salary,
  ROW_NUMBER() OVER (ORDER BY salary DESC), -- 1, 2, 3, 4
  RANK()       OVER (ORDER BY salary DESC), -- 1, 2, 2, 4
  DENSE_RANK() OVER (ORDER BY salary DESC)  -- 1, 2, 2, 3
FROM employees;
Q22 How does GROUP BY work? What is the ONLY_FULL_GROUP_BY mode? ▼

GROUP BY groups rows with the same values in specified columns. In ONLY_FULL_GROUP_BY mode (default in MySQL 5.7.5+), all columns in SELECT must either be in GROUP BY or be aggregated.

SELECT department, COUNT(*), AVG(salary)
FROM employees
GROUP BY department;

-- ❌ Error in ONLY_FULL_GROUP_BY: 'name' not in GROUP BY
SELECT department, name, COUNT(*) FROM employees GROUP BY department;

-- ✅ Fix: include name in GROUP BY or use ANY_VALUE()
SELECT department, ANY_VALUE(name), COUNT(*) FROM employees GROUP BY department;

-- GROUP BY with ROLLUP (subtotals)
SELECT department, SUM(salary) FROM employees
GROUP BY department WITH ROLLUP; -- adds grand total row
Q23 How do you write a query to find the second highest salary? ▼

A classic interview question with multiple approaches:

-- Method 1: Subquery (handles NULLs gracefully)
SELECT MAX(salary) AS second_highest
FROM employees
WHERE salary < (SELECT MAX(salary) FROM employees);

-- Method 2: LIMIT with OFFSET
SELECT DISTINCT salary
FROM employees
ORDER BY salary DESC
LIMIT 1 OFFSET 1;

-- Method 3: Dense rank (Nth highest generalized)
SELECT salary FROM (
  SELECT salary, DENSE_RANK() OVER (ORDER BY salary DESC) AS rnk
  FROM employees
) ranked WHERE rnk = 2 LIMIT 1;
Q24 How do you find duplicate rows in a table? ▼ 
-- Find duplicate emails
SELECT email, COUNT(*) AS occurrences
FROM users
GROUP BY email
HAVING COUNT(*) > 1;

-- Show full duplicate rows
SELECT * FROM users
WHERE email IN (
  SELECT email FROM users
  GROUP BY email HAVING COUNT(*) > 1
);

-- Delete duplicates, keeping lowest ID
DELETE u1 FROM users u1
INNER JOIN users u2
  ON u1.email = u2.email AND u1.id > u2.id;
Q25 What are string functions in MySQL? ▼ 
CONCAT('Hello', ' ', 'World')   -- 'Hello World'
CONCAT_WS(', ', 'Alice', 'Bob') -- 'Alice, Bob' (with separator)
LENGTH('hello')                   -- 5 (bytes)
CHAR_LENGTH('hello')              -- 5 (characters, UTF-8 safe)
UPPER('hello')                    -- 'HELLO'

TRIM('  hello  ')                 -- 'hello'
SUBSTRING('Hello World', 7, 5)   -- 'World'
REPLACE('Hello', 'l', 'r')       -- 'Herro'

LEFT('Hello', 3)                  -- 'Hel'
RIGHT('Hello', 3)                 -- 'llo'
LPAD('5', 3, '0')                 -- '005'
FORMAT(1234567.89, 2)            -- '1,234,567.89'
REGEXP_REPLACE(col, '[^a-z]', '') -- MySQL 8.0+
Q26 What are date/time functions in MySQL? ▼ 
NOW()                            -- '2025-05-27 14:30:00'
CURDATE()                        -- '2025-05-27'
CURTIME()                        -- '14:30:00'
DATE_FORMAT(NOW(), '%d %M %Y')  -- '27 May 2025'

🟡 Indexes & Keys — Q27 to Q36

Index types, how they work, and when to use them

Q27 What is an index in MySQL? How does it improve performance? ▼

An index is a data structure (B-Tree by default in InnoDB) that allows MySQL to find rows without scanning the entire table — like a book's index pointing to page numbers.

  • Without index: Full table scan — O(n) — reads every row
  • With index: B-Tree lookup — O(log n) — navigates the tree to the data
-- Create index
CREATE INDEX idx_email ON users(email);
CREATE UNIQUE INDEX idx_username ON users(username);
CREATE INDEX idx_dept_salary ON employees(department, salary); -- composite

-- View indexes on a table
SHOW INDEX FROM users;

-- Drop index
DROP INDEX idx_email ON users;
ALTER TABLE users DROP INDEX idx_email;
Indexes speed up reads but slow down writes (INSERT/UPDATE/DELETE must update index). Don't over-index.
Q28 What are the different types of indexes in MySQL? ▼
  • PRIMARY KEY: Clustered index. Table rows physically ordered by it. One per table. Always unique and NOT NULL.
  • UNIQUE INDEX: Ensures column values are unique. Allows one NULL.
  • INDEX (non-unique): Speeds up queries. Duplicates allowed.
  • COMPOSITE INDEX: Index on multiple columns. Order matters — leftmost prefix rule.
  • FULLTEXT INDEX: For text search using MATCH...AGAINST. Only on CHAR, VARCHAR, TEXT columns.
  • SPATIAL INDEX: For geometric data types (MyISAM, InnoDB 5.7+).
  • COVERING INDEX: Not a type — when an index contains all columns needed by the query (no table lookup needed).
Q29 What is the difference between clustered and non-clustered indexes? ▼
  • Clustered Index (PRIMARY KEY in InnoDB): Table rows are physically stored in index order. Each table can have ONLY ONE. Lookups are very fast since data IS the index leaf.
  • Non-clustered Index (Secondary Index): A separate structure that stores index key + pointer (primary key) to the actual row. Requires an extra lookup to get full row data ("bookmark lookup").
-- InnoDB: PRIMARY KEY is always the clustered index
-- Secondary index on email stores: email + id (PK)
-- Looking up by email: B-Tree lookup → get id → clustered lookup

-- Covering index avoids second lookup
SELECT id, email FROM users WHERE email = 'a@b.com';
-- If index on (email, id): all data from index alone!
Q30 What is a composite index and the leftmost prefix rule? ▼

A composite index covers multiple columns. MySQL can use this index for queries that start from the leftmost column(s) of the index definition — this is the leftmost prefix rule.

CREATE INDEX idx_dept_status_date ON employees(department, status, hire_date);

-- ✅ Uses index (starts from leftmost)
WHERE department = 'Engineering'
WHERE department = 'Engineering' AND status = 'active'
WHERE department = 'Engineering' AND status = 'active' AND hire_date > '2023-01-01'

-- ❌ Cannot use index (skips department)
WHERE status = 'active'
WHERE hire_date > '2023-01-01'
Design composite indexes with the most selective column first, and match your most common query patterns.
Q31 What is EXPLAIN and how do you use it? ▼

EXPLAIN shows how MySQL will execute a query — whether it uses indexes, how many rows it will scan, and which join type it will use. Essential for query optimization.

EXPLAIN SELECT * FROM users WHERE email = 'alice@example.com';

-- Key columns to check:
-- type:  ALL (full scan❌) | range | ref | eq_ref | const ✅
-- key:   which index is used (NULL = no index ❌)
-- rows:  estimated rows examined (lower = better)
-- Extra: 'Using filesort'❌ | 'Using index'✅ | 'Using where'

-- EXPLAIN ANALYZE (MySQL 8.0.18+) — actual execution stats
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'a@b.com';
Q32 When does MySQL NOT use an index? ▼

MySQL will skip an index in several situations:

-- 1. Function on indexed column
WHERE YEAR(created_at) = 2024       -- ❌ no index
WHERE created_at >= '2024-01-01'     -- ✅ uses index

-- 2. Type mismatch (implicit cast)
WHERE phone_number = 1234567890      -- ❌ phone is VARCHAR
WHERE phone_number = '1234567890'    -- ✅

-- 3. Leading wildcard
WHERE name LIKE '%son'              -- ❌ no index
WHERE name LIKE 'Ali%'              -- ✅ uses index

-- 4. OR without indexes on all columns
-- 5. Low selectivity (column with few distinct values)
-- 6. Very small tables (full scan is faster than index lookup)
Q33 What is a covering index? ▼

A covering index includes all columns required by a query so MySQL can satisfy the entire query from the index alone — without reading the actual table rows (no "table heap" lookup). This is shown as "Using index" in EXPLAIN.

CREATE INDEX idx_cover ON orders(user_id, status, amount, created_at);

-- This query is fully covered by the index above
SELECT status, amount, created_at
FROM orders
WHERE user_id = 42
  AND status = 'paid';
-- EXPLAIN shows: Using index (no table lookup needed!)
Covering indexes are one of the most powerful optimizations. Design them around your most expensive queries.
Q34 What is FULLTEXT search in MySQL? ▼

FULLTEXT indexes enable natural language text search — much more powerful than LIKE '%keyword%'. Supports relevance ranking, boolean mode, and phrase search.

ALTER TABLE articles ADD FULLTEXT INDEX idx_ft (title, content);

-- Natural language search (relevance ranked)
SELECT title, MATCH(title, content) AGAINST ('mysql performance') AS score
FROM articles
WHERE MATCH(title, content) AGAINST ('mysql performance')
ORDER BY score DESC;

-- Boolean mode: +required -excluded "exact phrase"
WHERE MATCH(title, content) AGAINST ('+mysql -oracle "query optimization"' IN BOOLEAN MODE);
Q35 What is a foreign key and what are ON DELETE / ON UPDATE actions? ▼

A foreign key enforces referential integrity between two tables. The ON DELETE / ON UPDATE clauses define what happens to child rows when the parent row is deleted or updated.

  • CASCADE: Delete/update child rows automatically
  • SET NULL: Set child FK column to NULL
  • RESTRICT / NO ACTION: Prevent delete/update if child rows exist (default)
  • SET DEFAULT: Set to default value (rarely used)
CREATE TABLE orders (
  id      INT PRIMARY KEY,
  user_id INT NOT NULL,
  FOREIGN KEY (user_id) REFERENCES users(id)
    ON DELETE CASCADE    -- delete orders when user deleted
    ON UPDATE CASCADE    -- update FK if user PK changes
);
Q36 What is index cardinality and selectivity? ▼
  • Cardinality: The number of unique values in an index. Higher = more selective.
  • Selectivity: Cardinality / Total rows. Closer to 1.0 = index is very effective.
-- Check cardinality
SHOW INDEX FROM users;
-- Cardinality column shows estimated unique values

-- High selectivity (good for index):
-- email (unique): selectivity = 1.0
-- user_id: selectivity ~0.9

-- Low selectivity (index rarely helps):
-- gender (M/F): selectivity = 2/1000000 = 0.000002
-- is_active (true/false): selectivity ~0.5

ANALYZE TABLE users; -- update index statistics

🟠 Stored Procedures, Functions & Triggers — Q37 to Q46

Server-side programming in MySQL

Q37 What is a Stored Procedure? How is it different from a function? ▼
  • Stored Procedure: A saved block of SQL code. Can have IN, OUT, INOUT parameters. Can return multiple result sets. Called with CALL. Can use DML, DDL, transactions.
  • Function: Must return a single value. Used inside SQL expressions (SELECT, WHERE). Cannot use transactions or modify data (strictly). Deterministic/non-deterministic.
-- Stored Procedure
DELIMITER //
CREATE PROCEDURE GetUserOrders(
  IN  p_user_id INT,
  OUT p_count   INT
)
BEGIN
  SELECT COUNT(*) INTO p_count FROM orders WHERE user_id = p_user_id;
  SELECT * FROM orders WHERE user_id = p_user_id;
END//
DELIMITER ;

CALL GetUserOrders(1, @cnt);
SELECT @cnt;
Q38 How do you create a user-defined function (UDF) in MySQL? ▼ 
DELIMITER //
CREATE FUNCTION FormatCurrency(amount DECIMAL(10,2), currency VARCHAR(3))
RETURNS VARCHAR(20)
DETERMINISTIC
BEGIN
  RETURN CONCAT(currency, ' ', FORMAT(amount, 2));

-- Use in a query
SELECT FormatCurrency(price, 'USD') FROM products;

-- DETERMINISTIC = same input always gives same output
-- NOT DETERMINISTIC = may use NOW(), RAND() etc.
Q39 What are control flow statements in stored procedures? ▼ 
DELIMITER //
CREATE PROCEDURE DemoFlow(p_score INT)
BEGIN
  -- IF / ELSEIF / ELSE
  IF p_score >= 90 THEN
    SELECT 'A';
  ELSEIF p_score >= 80 THEN
    SELECT 'B';
  ELSE
    SELECT 'C';
  END IF;

  -- CASE
  CASE p_score DIV 10
    WHEN 10 THEN SELECT 'Perfect';
    WHEN  9 THEN SELECT 'Excellent';
    ELSE        SELECT 'Good';
  END CASE;

  -- WHILE loop
  SET @i = 0;
  WHILE @i < 5 DO
    SET @i = @i + 1;
  END WHILE;

  -- LOOP with LEAVE
  my_loop: LOOP
    IF @i >= 10 THEN LEAVE my_loop; END IF;
    SET @i = @i + 1;
  END LOOP;
END//
DELIMITER ;
Q40 What is a Cursor in MySQL? ▼

A cursor allows you to iterate over a result set row by row inside a stored procedure. Always declare in this order: variables → cursor → handler.

DELIMITER //
CREATE PROCEDURE ProcessOrders()
BEGIN
  DECLARE done BOOLEAN DEFAULT FALSE;
  DECLARE order_id INT;
  DECLARE cur CURSOR FOR
    SELECT id FROM orders WHERE status = 'pending';
  DECLARE CONTINUE HANDLER FOR NOT FOUND SET done = TRUE;

  OPEN cur;
  read_loop: LOOP
    FETCH cur INTO order_id;
    IF done THEN LEAVE read_loop; END IF;
    -- process each order_id
    UPDATE orders SET status='processing' WHERE id=order_id;
  END LOOP;
  CLOSE cur;
END//
DELIMITER ;
Q41 What is a Trigger in MySQL? ▼

A trigger is a stored program that automatically executes (fires) in response to INSERT, UPDATE, or DELETE events on a table. Fires BEFORE or AFTER the event.

DELIMITER //
-- Audit log trigger: record changes to users table
CREATE TRIGGER after_user_update
AFTER UPDATE ON users
FOR EACH ROW
BEGIN
  INSERT INTO audit_log (table_name, record_id, old_val, new_val, changed_at)
  VALUES ('users', OLD.id, OLD.email, NEW.email, NOW());
END//

-- BEFORE INSERT: auto-set timestamps, validate
CREATE TRIGGER before_order_insert
BEFORE INSERT ON orders
FOR EACH ROW
BEGIN
  IF NEW.amount <= 0 THEN
    SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = 'Amount must be positive';
  END IF;
END//
DELIMITER ;
Q42 What is a View in MySQL? ▼

A view is a virtual table defined by a stored SELECT query. It doesn't store data — it runs the underlying query each time it's accessed. Useful for simplifying complex queries and controlling data access.

-- Create a view
CREATE VIEW active_user_orders AS
SELECT
  u.id, u.name, u.email,
  COUNT(o.id)    AS order_count,
  SUM(o.amount)  AS total_spent
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE u.status = 'active'
GROUP BY u.id;

-- Use like a table
SELECT * FROM active_user_orders WHERE total_spent > 1000;

-- Drop / replace
DROP VIEW active_user_orders;
CREATE OR REPLACE VIEW active_user_orders AS ...;
Q43 What is an Event (scheduled job) in MySQL? ▼

MySQL Events are tasks that run automatically on a schedule — like cron jobs inside the database. Requires the Event Scheduler to be enabled.

-- Enable event scheduler
SET GLOBAL event_scheduler = ON;

-- One-time event
CREATE EVENT cleanup_once
ON SCHEDULE AT NOW() + INTERVAL 1 HOUR
DO DELETE FROM temp_data WHERE created_at < NOW() - INTERVAL 7 DAY;

-- Recurring event
CREATE EVENT daily_cleanup
ON SCHEDULE EVERY 1 DAY
STARTS '2025-01-01 02:00:00'
DO
BEGIN
  DELETE FROM sessions WHERE expires_at < NOW();
  DELETE FROM logs WHERE created_at < NOW() - INTERVAL 30 DAY;
END;
Q44 What is the CASE expression in MySQL? ▼

CASE is an inline conditional expression — like if/else inside a query. Two forms: simple CASE (equality) and searched CASE (conditions).

-- Simple CASE
SELECT name,
  CASE status
    WHEN 'A' THEN 'Active'
    WHEN 'I' THEN 'Inactive'
    ELSE 'Unknown'
  END AS status_label
FROM users;

-- Searched CASE (conditions)
SELECT name, salary,
  CASE
    WHEN salary >= 100000 THEN 'Senior'
    WHEN salary >= 60000  THEN 'Mid-level'
    ELSE 'Junior'
  END AS level
FROM employees;

-- CASE in aggregate (pivot-style)
SELECT
  SUM(CASE WHEN status='paid'    THEN amount ELSE 0 END) AS paid_total,
  SUM(CASE WHEN status='pending' THEN amount ELSE 0 END) AS pending_total
FROM orders;
Q45 What are variables in MySQL? What is the difference between user variables and local variables? ▼
  • User-defined variables (@var): Session-scoped. Persist for the entire connection. Can be used outside procedures.
  • Local variables (DECLARE): Only exist inside a stored procedure/function. More strict typing.
  • System variables (@@var): Server-wide or session settings (@@max_connections, @@version).
-- User variable
SET @total = 0;
SELECT @total := SUM(amount) FROM orders;
SELECT @total;

-- Local variable (inside procedure only)
DECLARE v_count INT DEFAULT 0;
SELECT COUNT(*) INTO v_count FROM orders;

-- System variables
SELECT @@version, @@max_connections;
SET @@session.sql_mode = 'STRICT_ALL_TABLES';
Q46 What is the difference between a Stored Procedure and a Trigger? ▼
  • Stored Procedure: Called explicitly with CALL. Can accept parameters. Can return result sets. Run on demand.
  • Trigger: Fires automatically on DML events (INSERT/UPDATE/DELETE). Cannot be called directly. Cannot return result sets. Cannot accept parameters (uses OLD/NEW row data).
Use stored procedures for reusable business logic. Use triggers for automatic audit logging, validation, and maintaining derived data.

🔴 Transactions & Locks — Q47 to Q56

ACID properties, isolation levels, locking, and concurrency control

Q47 What is a transaction? What are ACID properties? ▼

A transaction is a sequence of SQL operations treated as a single unit of work — either all succeed or all fail.

  • Atomicity: All or nothing. If any step fails, the entire transaction rolls back.
  • Consistency: Transaction brings the database from one valid state to another. All constraints are satisfied.
  • Isolation: Concurrent transactions don't interfere with each other. Each sees a consistent snapshot.
  • Durability: Once committed, data is permanently stored even if the server crashes.
START TRANSACTION;
  UPDATE accounts SET balance = balance - 500 WHERE id = 1;
  UPDATE accounts SET balance = balance + 500 WHERE id = 2;
COMMIT;   -- both updates succeed
-- or
ROLLBACK; -- neither update applies
Q48 What are transaction isolation levels? ▼

Isolation levels control how concurrent transactions see each other's uncommitted data. Higher isolation = fewer anomalies, but more locking overhead.

  • READ UNCOMMITTED: Can read uncommitted changes (dirty reads). Lowest isolation, highest performance.
  • READ COMMITTED: Can only read committed data. No dirty reads. Non-repeatable reads possible.
  • REPEATABLE READ (default InnoDB): Same read within a transaction always returns same data. No dirty/non-repeatable reads. Phantom reads possible.
  • SERIALIZABLE: Highest isolation. Full serialization — transactions run as if sequential. Slowest.
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SET SESSION transaction_isolation = 'READ-COMMITTED';
SELECT @@transaction_isolation;
Q49 What are dirty reads, non-repeatable reads, and phantom reads? ▼
  • Dirty Read: Transaction A reads data that Transaction B has modified but not yet committed. If B rolls back, A has read invalid data.
  • Non-repeatable Read: Transaction A reads a row, Transaction B updates and commits it, Transaction A reads the same row and gets a different value.
  • Phantom Read: Transaction A queries a set of rows, Transaction B inserts a new matching row and commits, Transaction A re-queries and sees the new "phantom" row.
Isolation Level     | Dirty | Non-Repeatable | Phantom
READ UNCOMMITTED    |  ✅   |      ✅        |   ✅
READ COMMITTED      |  ❌   |      ✅        |   ✅
REPEATABLE READ     |  ❌   |      ❌        |   ✅ (InnoDB mostly ❌)
SERIALIZABLE        |  ❌   |      ❌        |   ❌
(✅ = possible, ❌ = prevented)
Q50 What is SAVEPOINT in transactions? ▼

A SAVEPOINT creates a named checkpoint within a transaction. You can roll back to a savepoint without rolling back the entire transaction — enabling partial rollbacks.

START TRANSACTION;

INSERT INTO orders (user_id, amount) VALUES (1, 100);
SAVEPOINT sp_after_order;

INSERT INTO payments (order_id, amount) VALUES (LAST_INSERT_ID(), 100);
-- Payment fails, but we want to keep the order
ROLLBACK TO SAVEPOINT sp_after_order;
-- Order is still inserted, payment is rolled back

RELEASE SAVEPOINT sp_after_order;
COMMIT;
Q51 What is locking in MySQL? What is the difference between row-level and table-level locking? ▼
  • Table-level locking (MyISAM): Locks the entire table. Simple but limits concurrency — only one writer at a time.
  • Row-level locking (InnoDB): Locks only the affected rows. Much better concurrency. Multiple writers on different rows simultaneously.
-- Explicit table lock
LOCK TABLES orders READ, products WRITE;
-- ... do work ...
UNLOCK TABLES;

-- Row-level: SELECT ... FOR UPDATE (pessimistic lock)
START TRANSACTION;
SELECT * FROM inventory WHERE product_id = 1 FOR UPDATE;
-- Row is locked until COMMIT/ROLLBACK
UPDATE inventory SET qty = qty - 1 WHERE product_id = 1;
COMMIT;
Q52 What is a deadlock? How does MySQL handle it? ▼

A deadlock occurs when two transactions each hold a lock that the other needs — a circular wait. Neither can proceed.

-- Deadlock example
-- Transaction A: Locks row 1, then tries to lock row 2
-- Transaction B: Locks row 2, then tries to lock row 1
-- → Deadlock!

MySQL's response: Automatically detects deadlocks and rolls back the transaction with the smallest undo log (least work done). The other transaction proceeds. The rolled-back transaction gets Error 1213.

Prevention: Always access resources in the same order across transactions. Keep transactions short. Use lower isolation levels where safe.

-- Check deadlocks
SHOW ENGINE INNODB STATUS; -- shows last deadlock
Q53 What is MVCC (Multi-Version Concurrency Control)? ▼

MVCC is how InnoDB provides consistent reads without locking. Instead of blocking readers when a writer holds a lock, InnoDB keeps multiple versions of rows (undo log). Readers see the version that was current when their transaction started.

  • Writers create a new version of the row and store the old version in the undo log
  • Readers read from a consistent snapshot (old version) without blocking
  • Result: Readers never block writers, writers never block readers
This is why InnoDB's REPEATABLE READ doesn't need to lock rows for SELECT — it uses snapshots from the undo log.
Q54 What is SELECT ... FOR UPDATE vs SELECT ... FOR SHARE? ▼
  • FOR UPDATE (exclusive lock): Locks selected rows — no other transaction can read (in locking mode) or write them. Use when you intend to update the rows.
  • FOR SHARE / LOCK IN SHARE MODE: Shared lock — other transactions can read but cannot write. Use when you want to ensure rows aren't changed while you read them.
-- Pessimistic locking: reserve a seat
START TRANSACTION;
SELECT * FROM seats WHERE id = 42 AND status = 'available'
FOR UPDATE;
-- Other transactions block here until we commit
UPDATE seats SET status = 'booked' WHERE id = 42;
COMMIT;
Q55 What is the difference between optimistic and pessimistic locking? ▼
  • Pessimistic Locking: Lock the row when reading it (FOR UPDATE). Prevents conflicts but reduces concurrency. Good when conflicts are frequent.
  • Optimistic Locking: Read without locking. Check at update time (using a version column) whether anyone else changed the row. Retry if conflict detected. Better concurrency, more complex logic.
-- Optimistic locking with version column
SELECT id, balance, version FROM accounts WHERE id = 1;
-- Got: balance=1000, version=5

UPDATE accounts
SET balance = 900, version = 6
WHERE id = 1 AND version = 5; -- fails if someone else updated

SELECT ROW_COUNT(); -- 0 = conflict, retry; 1 = success
Q56 What is autocommit in MySQL? ▼

In autocommit mode (default ON), every SQL statement is automatically committed as its own transaction. To group multiple statements, you must explicitly use START TRANSACTION ... COMMIT.

-- Check autocommit
SELECT @@autocommit; -- 1 = ON (default)

-- Disable for session
SET autocommit = 0;
-- Now every statement needs explicit COMMIT

-- DDL statements always auto-commit (cannot be rolled back)
CREATE TABLE test (id INT);   -- auto-commited even in transaction
DROP TABLE test;               -- auto-committed

🟣 Normalization & Database Design — Q57 to Q65

Normal forms, schema design, and relationships

Q57 What is database normalization? Why is it important? ▼

Normalization is the process of organizing a database to reduce data redundancy and improve data integrity. It involves decomposing tables into smaller, well-structured ones following specific rules (normal forms).

Benefits:

  • Eliminates data redundancy — no duplicate data
  • Prevents update anomalies — change data in one place
  • Reduces storage space
  • Improves data integrity and consistency

Trade-off: More normalized = more JOINs needed = potentially slower reads. Data warehouses often intentionally denormalize for read performance.

Q58 Explain the First, Second, and Third Normal Forms (1NF, 2NF, 3NF). ▼
  • 1NF (First Normal Form): Each column has atomic (indivisible) values. No repeating groups or arrays. Each row is unique (has a primary key).
  • 2NF (Second Normal Form): Must be in 1NF + every non-key column depends on the ENTIRE primary key (no partial dependencies). Mainly applies to composite keys.
  • 3NF (Third Normal Form): Must be in 2NF + no transitive dependencies (non-key columns must not depend on other non-key columns).
-- 1NF violation: multiple values in one cell
-- phones: '555-1234, 555-5678' ❌ → separate rows ✅

-- 2NF violation (composite PK: order_id + product_id)
-- product_name depends only on product_id (partial) ❌
-- Solution: move product_name to products table ✅

-- 3NF violation: zip → city (transitive)
-- zip_code → city depends on zip, not on user_id ❌
-- Solution: move to zip_codes table ✅
Q59 What is BCNF (Boyce-Codd Normal Form)? ▼

BCNF is a stronger version of 3NF. A table is in BCNF if for every functional dependency X → Y, X must be a superkey (a candidate key or superset of a candidate key). In practice, 3NF is usually sufficient for most applications.

Most interview questions stop at 3NF. Mention BCNF to show deeper knowledge, but acknowledge that practical database design often balances normalization with performance.
Q60 What are the types of relationships in a relational database? ▼
  • One-to-One (1:1): One row in A maps to exactly one row in B. Implemented via shared primary key or unique foreign key. Used to split optional or sensitive columns.
  • One-to-Many (1:N): One row in A maps to many rows in B. Most common. Implemented via foreign key in the "many" table.
  • Many-to-Many (M:N): Many rows in A map to many rows in B. Requires a junction/bridge table with two foreign keys.
-- One-to-Many: one user, many orders
orders.user_id → users.id

-- Many-to-Many: students ↔ courses
CREATE TABLE student_courses (
  student_id INT REFERENCES students(id),
  course_id  INT REFERENCES courses(id),
  enrolled_at DATE,
  PRIMARY KEY (student_id, course_id)
);
Q61 What is denormalization and when would you use it? ▼

Denormalization intentionally introduces redundancy into a normalized schema to improve read performance — trading write efficiency and storage for faster queries.

When to use:

  • Read-heavy applications where JOIN performance is a bottleneck
  • Data warehouses / analytics / reporting (OLAP)
  • Aggregate columns that are expensive to compute each time
-- Denormalized: store total_orders on user row
ALTER TABLE users ADD COLUMN order_count INT DEFAULT 0;

-- Maintain via trigger
CREATE TRIGGER after_order_insert
AFTER INSERT ON orders FOR EACH ROW
UPDATE users SET order_count = order_count + 1
WHERE id = NEW.user_id;
Q62 What is a surrogate key vs a natural key? ▼
  • Natural key: A key derived from the real-world data (email, SSN, ISBN). Meaningful but may change, be long, or not be guaranteed unique.
  • Surrogate key: An artificial key with no business meaning (AUTO_INCREMENT id, UUID). Stable, simple, and independent of business data. Preferred in most cases.
-- Natural key (email can change!)
CREATE TABLE users (
  email VARCHAR(255) PRIMARY KEY  -- ❌ risky
);

-- Surrogate key (preferred)
CREATE TABLE users (
  id    INT AUTO_INCREMENT PRIMARY KEY,  -- ✅ stable
  email VARCHAR(255) UNIQUE NOT NULL
);
Q63 What is a composite key? ▼

A composite key is a primary key that consists of two or more columns. Used when no single column uniquely identifies a row, but a combination of columns does — most common in junction tables for many-to-many relationships.

CREATE TABLE order_items (
  order_id   INT NOT NULL,
  product_id INT NOT NULL,
  quantity   INT NOT NULL,
  price      DECIMAL(10,2) NOT NULL,
  PRIMARY KEY (order_id, product_id),     -- composite PK
  FOREIGN KEY (order_id)   REFERENCES orders(id),
  FOREIGN KEY (product_id) REFERENCES products(id)
);
Q64 What are database design best practices? ▼
  • Always define a PRIMARY KEY on every table
  • Use surrogate keys (AUTO_INCREMENT) over natural keys
  • Use appropriate data types — DECIMAL for money, TIMESTAMP for auto-timezone, VARCHAR not CHAR unless fixed
  • Use NOT NULL wherever possible — avoid nullable columns
  • Add indexes on foreign keys and frequently queried columns
  • Use FOREIGN KEY constraints to enforce referential integrity
  • Keep table and column names lowercase, snake_case, descriptive
  • Add created_at and updated_at timestamp columns to every table
  • Avoid storing calculated values — compute them at query time (or use generated columns)
  • Use ENUM for small, fixed sets; use a lookup table for growing sets
Q65 What are Generated (Computed) Columns in MySQL? ▼

Generated columns (MySQL 5.7+) are columns whose values are computed from other columns automatically. Two types: VIRTUAL (computed on read) and STORED (persisted on write).

CREATE TABLE products (
  id         INT PRIMARY KEY,
  price      DECIMAL(10,2),
  tax_rate   DECIMAL(5,4) DEFAULT 0.18,
  price_incl_tax DECIMAL(10,2)
    AS (price * (1 + tax_rate)) VIRTUAL,   -- computed on read
  full_name  VARCHAR(200),
  name_upper VARCHAR(200)
    AS (UPPER(full_name)) STORED           -- persisted, indexable
);

-- Can index STORED generated columns
CREATE INDEX idx_name_upper ON products(name_upper);

⚫ Performance & Advanced — Q66 to Q77

Query optimization, replication, partitioning, JSON, and MySQL 8.0+ features

Q66 How do you optimize a slow MySQL query? ▼

Step-by-step optimization process:

  • 1. Enable slow query log — identify slow queries
  • 2. Run EXPLAIN — find missing indexes, full table scans
  • 3. Add indexes — on WHERE, JOIN, ORDER BY columns
  • 4. Rewrite query — avoid SELECT *, use covering indexes, replace subqueries with JOINs
  • 5. Avoid functions on indexed columns — prevents index use
  • 6. Analyze/optimize tables — update statistics
  • 7. Cache results — Redis for frequently read, rarely changed data
  • 8. Consider partitioning — for very large tables
-- Enable slow query log
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1; -- log queries > 1 second

-- Update table statistics
ANALYZE TABLE orders;
OPTIMIZE TABLE orders; -- defrag, reclaim space
Q67 What is query caching in MySQL? ▼

The MySQL Query Cache stored the text of SELECT statements and their results — identical queries were served from cache without re-executing. However, it was deprecated in MySQL 5.7.20 and removed in MySQL 8.0 because it caused more harm than good under high concurrency (global mutex on cache writes/invalidations).

Modern alternatives:

  • Application-level caching: Redis, Memcached — cache query results in your app layer
  • MySQL's InnoDB Buffer Pool — caches data pages and indexes in memory
  • ProxySQL: Query routing proxy with caching capabilities
Q68 What is MySQL Replication? What are the types? ▼

Replication copies data from one MySQL server (source/primary) to one or more servers (replicas/secondaries) in near real-time. Used for read scaling, high availability, and backup.

  • Async Replication (default): Source doesn't wait for replica to confirm. Fastest but replica may lag. Risk of data loss on source crash.
  • Semi-synchronous: Source waits for at least one replica to acknowledge before committing. Safer, slightly slower.
  • Group Replication (MySQL 5.7.17+): Multi-master, fully synchronous, built-in failover. Powers InnoDB Cluster.
-- Statement-based vs Row-based vs Mixed binlog format
SET GLOBAL binlog_format = 'ROW'; -- recommended

-- Check replication status
SHOW REPLICA STATUS\G
SHOW MASTER STATUS;
Q69 What is table partitioning in MySQL? ▼

Partitioning splits a large table into smaller physical pieces (partitions) while appearing as one table to queries. Improves performance for queries that can prune partitions and makes archiving/deletion of old data fast.

-- RANGE partitioning by year
CREATE TABLE orders (
  id         INT NOT NULL,
  created_at DATE NOT NULL,
  amount     DECIMAL(10,2),
  PRIMARY KEY (id, created_at)
)
PARTITION BY RANGE (YEAR(created_at)) (
  PARTITION p2022 VALUES LESS THAN (2023),
  PARTITION p2023 VALUES LESS THAN (2024),
  PARTITION p2024 VALUES LESS THAN (2025),
  PARTITION p_future VALUES LESS THAN MAXVALUE
);

-- Drop old partition (instant, no DELETE overhead)
ALTER TABLE orders DROP PARTITION p2022;
Q70 How does MySQL handle JSON data?

MySQL 5.7.8+ has a native JSON data type that validates and stores JSON efficiently in binary format. MySQL 8.0 added multi-valued indexes on JSON arrays.

CREATE TABLE products (
  id   INT PRIMARY KEY,
  meta JSON
);

INSERT INTO products VALUES (
  1, '{"color":"red","tags":["sale","featured"],"weight":1.5}'
);

-- Access JSON fields
SELECT
  meta->'$.color'          AS color,      -- with quotes: "red"
  meta->>'$.color'         AS color_raw,  -- unquoted: red
  JSON_EXTRACT(meta, '$.weight') AS weight
FROM products;

-- Modify JSON
UPDATE products
SET meta = JSON_SET(meta, '$.color', 'blue', '$.size', 'large')
WHERE id = 1;

-- Index JSON array values (MySQL 8.0)
CREATE INDEX idx_tags ON products ((CAST(meta->'$.tags' AS CHAR(50) ARRAY)));
Q71 What is the InnoDB Buffer Pool? ▼

The InnoDB Buffer Pool is the most important MySQL performance knob — an in-memory cache that stores data pages and index pages. Reads/writes go through the buffer pool first, avoiding disk I/O.

-- Check buffer pool size
SELECT @@innodb_buffer_pool_size / 1024/1024/1024 AS GB;

-- Set to 70-80% of available RAM (in my.cnf)
-- innodb_buffer_pool_size = 12G

-- Monitor buffer pool usage
SHOW STATUS LIKE 'Innodb_buffer_pool%';
-- Innodb_buffer_pool_read_requests / Innodb_buffer_pool_reads
-- Hit rate = 1 - (reads / read_requests) → aim for > 99%
The single most impactful MySQL configuration change is setting innodb_buffer_pool_size to 70-80% of available RAM.
Q72 What is the difference between OLTP and OLAP? How does MySQL serve each? ▼
  • OLTP (Online Transaction Processing): Many short, concurrent read/write transactions. Row-oriented. MySQL InnoDB is optimized for this.
  • OLAP (Online Analytical Processing): Complex analytical queries over large datasets. Column-oriented. MySQL can serve it but dedicated tools (ClickHouse, Snowflake, BigQuery) are better.

MySQL 8.0 improvements for analytics: CTEs, Window Functions, Hash joins, invisible indexes, and better query optimizer.

Q73 What is INSERT ... ON DUPLICATE KEY UPDATE? ▼

An "upsert" — inserts a new row, or if a duplicate key conflict occurs, updates the existing row. More efficient than separate SELECT + INSERT or UPDATE.

-- Upsert: insert or update page view count
INSERT INTO page_views (page_url, view_count, last_viewed)
VALUES ('/home', 1, NOW())
ON DUPLICATE KEY UPDATE
  view_count = view_count + 1,
  last_viewed = NOW();

-- MySQL 8.0.20+: aliases for new values
INSERT INTO users (id, email, name) VALUES (1, 'a@b.com', 'Alice')
AS new_row
ON DUPLICATE KEY UPDATE
  email = new_row.email,
  name  = new_row.name;
Q74 What are invisible indexes in MySQL 8.0? ▼

Invisible indexes (MySQL 8.0+) are maintained by MySQL but ignored by the query optimizer — as if they don't exist. This lets you test removing an index without actually dropping it, and roll back instantly if performance degrades.

-- Make index invisible (optimizer ignores it)
ALTER TABLE users ALTER INDEX idx_email INVISIBLE;

-- Test queries — do they slow down?
EXPLAIN SELECT * FROM users WHERE email = 'a@b.com';

-- Restore if needed
ALTER TABLE users ALTER INDEX idx_email VISIBLE;

-- Force optimizer to see invisible indexes (for testing)
SET SESSION optimizer_switch='use_invisible_indexes=on';
Q75 What is the MySQL EXPLAIN FORMAT=JSON and how is it different from regular EXPLAIN? ▼

Regular EXPLAIN shows a tabular output. EXPLAIN FORMAT=JSON provides a much richer, tree-structured output with additional information like cost estimates, loop iterations, and attached conditions.

EXPLAIN FORMAT=JSON
SELECT u.name, COUNT(o.id)
FROM users u JOIN orders o ON u.id = o.user_id
GROUP BY u.id\G

-- JSON output includes:
-- "cost_info": { "query_cost": "125.45" }
-- "rows_examined_per_scan": 1
-- "rows_produced_per_join": 100
-- "filtered": "100.00"
-- "using_index": true
Q76 What are Roles in MySQL 8.0? ▼

Roles (MySQL 8.0+) are named collections of privileges that can be granted to users — making permission management much cleaner for large teams.

-- Create roles
CREATE ROLE 'app_read', 'app_write', 'app_admin';

-- Grant privileges to roles
GRANT SELECT ON mydb.* TO 'app_read';
GRANT INSERT, UPDATE, DELETE ON mydb.* TO 'app_write';
GRANT ALL ON mydb.* TO 'app_admin';

-- Grant roles to users
GRANT 'app_read', 'app_write' TO 'api_user'@'%';
GRANT 'app_admin' TO 'dba_user'@'localhost';

-- Activate roles for session
SET ROLE ALL;
SET ROLE 'app_read';

Query language:

  • Window functions (ROW_NUMBER, RANK, LAG, LEAD, etc.)
  • Common Table Expressions (WITH clause), including Recursive CTEs
  • Lateral derived tables
  • Hash joins (faster for many join scenarios)

Indexes:

  • Invisible indexes
  • Descending indexes
  • Multi-valued indexes (on JSON arrays)
  • Functional indexes (INDEX (UPPER(name)))

Other improvements:

  • Roles for user management
  • Default charset changed to utf8mb4
  • Atomic DDL — DDL operations are transactional
  • Improved JSON support (JSON_TABLE, multi-value indexes)
  • EXPLAIN ANALYZE
  • Better optimizer with cost model improvements
  • InnoDB enhancements (instant ADD COLUMN, etc.)
